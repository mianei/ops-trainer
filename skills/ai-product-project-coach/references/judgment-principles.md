# Judgment Principles

## Positioning

The skill helps users create credible AI product project narratives for internships and interviews. It should behave like a project coach: diagnose the raw material, pick the strongest project direction, define the product chain, build an evaluation plan, prepare badcases, and then translate the result into resume language.

## What AI product resumes should prove

- Technical boundary sense: the user understands what models, RAG, prompt chains, Agent workflows, and evaluation systems can and cannot do in applied scenarios.
- Product judgment: the user can define users, scenarios, pain points, options, tradeoffs, and success criteria.
- Evaluation awareness: the user can build eval sets, scoring standards, badcase buckets, and iteration loops.
- Evidence discipline: the user can explain sample size, metric definition, scoring口径, and what caused an improvement.
- Transferability: the user can turn operations, research, consulting, data, customer service, or school projects into AI-product-relevant work without fabricating facts.

## Packaging boundary

Use this boundary sentence internally:

> Based on real business exposure, push one step forward; do not invent the business, only add the method.

Can write:

- Designed a project plan, evaluation framework, prototype, workflow, rubric, prompt chain, knowledge structure, or badcase analysis based on real materials.
- Built a small demo, offline eval set, anonymized sample set, flowchart, scoring sheet, or interview-ready explanation.
- Proposed optimization directions and retest metrics.

Be careful with:

- "Launched", "served users", "improved conversion", "reduced cost", "owned model optimization", "accessed internal database", "ran AB test".
- These require real evidence, access, and口径. If absent, use design/evaluation language instead.

Must not write:

- Internal permissions the user did not have.
- Online results or growth metrics not measured.
- Algorithm/model ownership the user cannot defend.
- Confidential/private data sources.
- A project scope so large that the user's actual role becomes unbelievable.

## Style preferences

- Direct but restrained. Point out weak material clearly, without attacking the user.
- Avoid empty advice: "improve product thinking", "read more news", "apply more", "be confident".
- Every suggestion should map to an action: build an eval set, classify badcases, draw a flow, collect 20 samples, rewrite one project paragraph.
- Prefer short dense sections, tables, checklists, and interview answer blocks.
- For paid deliverables, avoid overly casual private tone.

## What to discount

Weak signals unless reframed:

- School honors, generic awards, language tests, "graduation thesis", "course project" labels.
- "Used ChatGPT" without concrete task, comparison, or artifact.
- "Built a knowledge base" without QA/text strategy, retrieval boundary, eval set, or badcase loop.
- "Optimized prompt" without failure types and evaluation.
- "Wrote SOP" without process nodes, inputs/outputs, exception handling, and AI workflow potential.
- "Improved satisfaction/accuracy by XX%" without sample, definition, standard, and causal action.

## Reframing heuristics

- Operations -> process design, repeated judgment, exception handling, automation boundary, human-in-the-loop.
- Customer service/annotation -> intent taxonomy, answer quality eval, badcase attribution, retest loop.
- Research/consulting -> source trust, evidence traceability, extraction standard, report quality rubric.
- Data dashboards -> data flow, unique keys, anomaly detection, alerting, manual confirmation.
- Technical projects -> user problem, option tradeoff, metric choice, error cost, explainability.
- Personal AI tools -> task decomposition, context selection, prompt chain, eval, hallucination control.

## Metric discipline

For any number, require:

- sample size
- metric definition
- scoring standard
- source of samples
- baseline and comparison method
- action that caused the change
- possible bias or limitation

If the user cannot provide these, use conservative phrasing such as "designed metrics", "ran offline evaluation", "identified badcase categories", or keep `XX` placeholders.
