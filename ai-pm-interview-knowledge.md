# 第一篇：LLM 基础与核心技术

## 1.1 LLM 是什么

大语言模型（Large Language Model）是通过海量文本数据训练出来的神经网络，核心能力是**预测下一个 token**。给定一段文本，模型预测最可能的下一个词，不断重复这个过程就生成了完整回答。

**关键认知**：LLM 不是"理解"语言，而是"统计上最合理地续写"。它没有真正的理解能力，但续写能力的涌现让它看起来像理解了。

### Token

Token 是 LLM 处理文本的最小单位。一个中文字通常等于 1-2 个 token，一个英文单词通常等于 1-3 个 token。"我是产品经理"约 5-6 个 token，"Hello World"约 2 个 token。

**产品意义**：Token 是计费单位，也是上下文窗口的计量单位。控制 prompt 长度就是控制 token 消耗。

### 上下文窗口

上下文窗口是模型一次能"看到"的最大 token 数。超过窗口的内容会被截断或遗忘。

| 模型 | 上下文窗口 |
|------|----------|
| GPT-3.5 | 4K / 16K |
| GPT-4 | 8K / 32K / 128K |
| Claude 3 | 200K |
| Gemini 1.5 Pro | 1M |

**产品意义**：上下文窗口不是越大越好——窗口越大，推理越慢、越贵。产品设计上要控制每次对话的上下文长度，而不是无脑塞满。

## 1.2 LLM 的训练流程

### 预训练（Pre-training）

在海量互联网文本上让模型学会"预测下一个词"。GPT-3 用了约 45TB 文本。产出的是"基础模型"——它知道很多事实，但不知道怎么和人对话。

### 监督微调（SFT）

用高质量"问题-回答"对教模型怎么对话。模型学会了对话格式、礼貌用语、拒绝不合适请求。如果你用 API，这部分是模型厂商做的。

### RLHF（人类反馈强化学习）

让人类对模型的多条回答排序，用这个排序信号训练奖励模型，再用奖励模型优化 LLM。RLHF 让模型更"对齐"人类偏好，但也可能让模型变得过于保守。

### 训练全景

```
互联网文本 → 预训练 → 基础模型（会续写，不会对话）
高质量对话数据 → SFT → 对话模型（会对话，但不够好）
人类偏好排序 → RLHF → 对齐模型（更有用、更安全）
```

## 1.3 RAG vs 微调

| | RAG | 微调（Fine-tuning） |
|------|-----|------|
| 改什么 | 改输入（检索外部知识注入） | 改模型（更新模型参数） |
| 知识更新 | 更新知识库即可 | 需要重新训练 |
| 成本 | 检索 + 推理 | 训练 GPU 成本 |
| 可解释性 | 高，知道从哪检索的 | 低，不知道模型怎么学的 |
| 幻觉控制 | 有事实锚点 | 依赖训练数据质量 |

**用 RAG**：知识频繁变化、需要引用来源、有明确知识库。

**用微调**：需要模型学会特定风格或推理模式、领域术语非常特殊。

大部分场景用 RAG 就够了。微调是试了所有 prompt 和 RAG 优化后还是不行的最后手段。

## 1.4 模型推理与部署

### 推理关键指标

| 指标 | 说明 | 产品意义 |
|------|------|---------|
| 首 token 延迟 | 从请求到第一个 token 出现的时间 | 用户感知的"响应速度" |
| 生成速度 | 每秒生成多少 token | 长回答的等待时间 |
| 吞吐量 | 每秒处理多少请求 | 并发用户数上限 |
| 成本 | 每 1K token 的价格 | 产品定价和利润 |

### 推理优化技术

| 技术 | 是什么 | 产品意义 |
|------|--------|---------|
| 量化 | 把模型参数精度降低（FP16→INT4） | 更快更便宜，精度可能下降 |
| 缓存 | 相同请求复用结果 | 高频问题零成本响应 |
| 流式输出 | 逐 token 输出 | 用户感知更快 |
| 批处理 | 多个请求合并处理 | 离线场景降成本 |

### 模型选择的产品判断

选模型不是选最好的，是选最合适的：任务难度、延迟要求、成本预算、数据安全、语言能力。

## 1.5 Prompt 的底层原理

LLM 的本质是"给定上文，预测下文"。Prompt 就是人为构造的"上文"，引导模型往你期望的方向续写。Prompt 不是"命令"，是"引子"。

### In-Context Learning（上下文学习）

LLM 能从 Prompt 中的例子"学会"怎么做，而不需要更新参数。Few-shot Prompt 就是利用这个能力——给几个例子，模型就能学会你想要的格式和逻辑。

### 为什么 Prompt 有时候不灵

- **位置偏差**：模型对 Prompt 开头和结尾更关注，中间易被忽略
- **长度偏差**：Prompt 越长，模型越容易"走神"
- **指令冲突**：多条矛盾指令，模型不知道听谁的
- **格式干扰**：Prompt 格式混乱，输出也跟着乱

## 1.6 多模态模型

| 能力 | 代表模型 | 应用场景 |
|------|---------|---------|
| 文生文 | GPT-4, Claude | 对话、写作、分析 |
| 文生图 | DALL-E, Midjourney | 设计、营销素材 |
| 图生文 | GPT-4V, Claude 3 | 图片分析、OCR |
| 文生音 | ElevenLabs | 有声内容、语音助手 |
| 音生文 | Whisper | 语音转文字 |
| 文生视频 | Sora, Runway | 视频创作 |

**产品判断**：用户输入是图片/音频→必须用多模态；纯粹文字场景→不用，成本更高。

# 第二篇：AI Agent 全景

## 2.1 Agent 基础概念

Agent（智能体）是一个能感知环境、自主决策、执行动作的 AI 系统。核心特征是可以**拆解任务、调用工具、多步推理、记忆上下文**。

**类比**：传统 LLM = 答题机器，你问它答，答完就忘。Agent = 实习生，能理解你要什么、自己拆步骤、需要时查资料、做完了汇报。

### Agent 的核心四要素

| 要素 | 做什么 | 举例 |
|------|--------|------|
| 规划（Planning） | 把大任务拆成小步骤 | "帮我做竞品分析" → 先搜竞品列表 → 逐个查信息 → 对比 → 输出报告 |
| 记忆（Memory） | 记住上下文和历史 | 用户 10 分钟前说过"我是产品实习生"，Agent 后续回答始终基于这个身份 |
| 工具（Tools） | 调用外部能力 | 搜索、读文件、发邮件、操作飞书文档 |
| 行动（Action） | 执行具体操作 | 调搜索 API → 读网页 → 写文件 → 返回结果 |

### Agent 和传统 Chatbot 的区别

| | Chatbot | Agent |
|------|---------|-------|
| 交互模式 | 一问一答 | 多步推理，自主执行 |
| 上下文 | 单轮或有限多轮 | 长期记忆 + 短期记忆 |
| 能力边界 | 纯文本生成 | 调用工具、读写文件、操作外部系统 |
| 任务复杂度 | 简单问答 | 复杂任务拆解和执行 |

## 2.2 Agent 架构模式

### Fixed Workflow（固定工作流）

预定义的、不可变的执行步骤链。每一步做什么是写死的。

**适用**：RAG 问答、报告生成、面试评分。**优点**：稳定、可预测、易调试、成本低。**缺点**：不灵活。

**产品判断**：能用 Fixed Workflow 搞定的，不要上 Agent。架构越简单越可靠。

### Sub-agent（子 Agent）

主 Agent 把复杂子任务派发给专门的小 Agent，子 Agent 独立完成后返回结果。

**适用**：子任务独立性强、可并行。**优点**：并行提效、职责分离。**缺点**：调度复杂、通信有损耗。

**产品判断**：不要为了"听起来高级"用 Sub-agent。如果子任务有明显先后依赖，串行比并行更可控。

### Agent Team（Agent 团队）

多个 Agent 各司其职，像一个团队协作完成任务。

**适用**：任务需要不同角色协同。**优点**：模拟真实协作。**缺点**：调度复杂、通信开销大、调试困难。

**产品判断**：Agent Team 是最容易被面试官挑战的架构。除非场景确实需要多角色协作，否则不要用。

### 架构选型决策树

```
任务是否固定不变？
  ├─ 是 → Fixed Workflow
  └─ 否 → 任务是否可以拆成独立子任务？
           ├─ 是，且子任务可并行 → Sub-agent
           ├─ 是，且子任务需要不同角色 → Agent Team
           └─ 否 → 单 Agent + 工具调用
```

## 2.3 RAG（检索增强生成）

### 核心流程

```
用户提问 → ①检索（从知识库找相关文档）→ ②增强（拼接检索结果到 Prompt）→ ③生成（LLM 基于拼接后的 Prompt 回答）
```

**为什么需要 RAG**：LLM 训练数据有截止日期、会幻觉、没见过企业私有知识。

### 检索策略对比

| 策略 | 原理 | 擅长 | 不擅长 |
|------|------|------|--------|
| 关键词检索（BM25） | 匹配关键词，算词频 | 精确匹配专业术语 | 同义词、语义相近但措辞不同 |
| 语义向量检索 | 文本转向量，算余弦相似度 | 理解"用户增长"≈"DAU 提升" | 精确匹配时噪声大 |
| 混合检索（Hybrid） | BM25 + 语义向量合并 | 兼顾精确和语义 | 实现复杂度稍高 |
| 多路召回 + 重排序 | 多路粗召回 → 精排重排 | 准确率最高 | 延迟和成本更高 |

### RAG 选型的产品判断

- 知识库 < 100 篇 → 关键词检索就够了
- 知识库 100-1000 篇 → 混合检索（BM25 + 语义）
- 知识库 > 1000 篇 → 多路召回 + 重排序
- 知识频繁更新 → 实时索引

### RAG 常见问题与优化

| 问题 | 现象 | 优化方向 |
|------|------|---------|
| 检索不到 | 知识库有答案但搜不出来 | 优化 chunk 策略、调整 embedding 模型 |
| 检索噪声 | 搜出大量不相关内容 | 加阈值过滤、重排序 |
| 上下文过长 | 拼接后超 token 限制 | 压缩检索结果 |
| 信息冲突 | 多个文档矛盾 | 加来源标注、时效性权重 |
| 答案被检索结果绑架 | 检索到垃圾内容模型照搬 | 加验证步骤 |

## 2.4 Prompt Engineering

### Prompt 的四层解耦结构

```
第一层：角色（Role）——你是谁
第二层：知识（Knowledge）——你知道什么
第三层：任务（Task）——你要做什么
第四层：格式（Format）——你怎么输出
```

**解耦的价值**：每层独立修改，改输出格式不碰评分逻辑；出问题能定位到具体层级；团队协作时不同人负责不同层。

### 常见 Prompt 技巧

| 技巧 | 说明 | 示例 |
|------|------|------|
| Few-shot | 给几个例子，让模型照猫画虎 | 给出 3 个"好回答 + 评分"的例子 |
| Chain-of-Thought | 让模型"一步步思考" | "请先分析用户回答的要点，再逐项评分" |
| 正反例 | 告诉模型什么是对什么是错 | "好：具体到场景。差：空泛的套话" |
| 约束边界 | 明确什么不能做 | "不要给职业建议，不要安抚情绪" |
| 结构化输出 | 要求 JSON/Markdown 格式 | "请严格按 JSON 格式输出" |

### Prompt 迭代的产品方法论

```
发现问题 → 定位到具体 Prompt 层 → 改前跑基线 → 改后跑评测 → 对比分数 → 决定保留/回滚
```

**一条规则**：一次只改一个变量。不要同时改角色定义和评分标准。

## 2.5 Agent 评测

### 评测金字塔

```
         ┌──────────────┐
         │  用户行为指标  │  ← 最难评，但最有价值
         │  （留存、复访） │
         ├──────────────┤
         │  端到端体验指标 │  ← 任务完成率、用户满意度
         │ （任务成功率）  │
         ├──────────────┤
         │  链路节点指标   │  ← 检索准确率、工具调用成功率
         │（检索、工具调用）│
         ├──────────────┤
         │  基础能力指标   │  ← 格式正确率、响应时间
         │ （格式、延迟）  │
         └──────────────┘
```

### 评测集设计：MECE 原则

维度之间互不重叠、共同穷尽。以面试评分 Agent 为例：

| 维度 | 评什么 | 不评什么 |
|------|--------|---------|
| 内容相关性 | 回答是否在回答问题 | 不评表达好不好 |
| 逻辑结构 | 是否有清晰的因果/递进 | 不评内容是否深刻 |
| 表达清晰度 | 是否简洁准确无歧义 | 不评逻辑是否严密 |
| 岗位匹配度 | 是否体现岗位能力 | 不评表达是否流畅 |
| 深度与细节 | 是否有具体案例和数据 | 不评是否匹配岗位 |

### 评测执行方式

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 人工标注 | 人打分，作为黄金标准 | 建立基线、校准模型 |
| LLM-as-a-Judge | 用 LLM 打分 | 规模化评测、快速迭代 |
| 规则检查 | 用正则/代码检查格式 | 输出格式校验 |
| A/B 测试 | 分流对比 | 已上线产品、有真实用户 |

### 评测指标速查

| 指标 | 定义 | 适用场景 |
|------|------|---------|
| 任务完成率 | 用户意图是否被满足 | 端到端评测 |
| 意图理解准确率 | Agent 是否正确理解用户意图 | 意图识别模块 |
| 工具调用准确率 | 工具选择是否正确、参数是否准确 | 工具调用模块 |
| 检索命中率 | 检索结果是否包含正确答案 | RAG 检索节点 |
| 幻觉率 | 输出中虚构信息的比例 | 生成质量评测 |
| 格式正确率 | 输出是否符合预期格式 | 输出解析 |
| 用户满意度 | 用户主观评分 | 上线后追踪 |
| 多轮对话成功率 | 多轮后是否完成目标 | 对话类 Agent |

## 2.6 Agent 记忆与上下文

### 记忆类型

| 类型 | 存什么 | 多久 | 举例 |
|------|--------|------|------|
| 短期记忆 | 当前对话历史 | 一次会话 | 用户刚才说了什么 |
| 长期记忆 | 用户画像、偏好 | 跨会话 | 用户是产品实习生 |
| 工作记忆 | 当前任务的状态 | 一个任务周期 | 正在搜第 3 个竞品 |

### 上下文窗口管理策略

| 策略 | 做法 | 优缺点 |
|------|------|--------|
| 滑动窗口 | 只保留最近 N 轮对话 | 简单，但丢失早期信息 |
| 摘要压缩 | 对早期对话做摘要 | 保留要点，但摘要可能失真 |
| 选择性遗忘 | 根据重要性决定保留什么 | 最优，但判断"重要性"本身很难 |
| 分层记忆 | 短期+长期+工作记忆分层管理 | 最接近人类记忆，架构复杂 |

### 产品判断：记忆设计的核心问题

1. 用户回来时 Agent 还记得他吗？记不住→每次重新介绍→体验差
2. 记住的信息会过期吗？用户换了岗位 Agent 还在用旧信息→体验更差
3. 记忆会不会泄露？用户 A 的对话被用来回答用户 B→这是事故

## 2.7 工具调用

### 工具设计原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 一个工具只做一件事 |
| 明确输入输出 | 参数名、类型、必填/可选、返回值结构都要清楚 |
| 错误处理 | 工具调用失败时 Agent 要知道怎么处理（重试/降级/报告） |
| 幂等性 | 同样的输入多次调用，结果应该一致 |
| 超时控制 | 外部 API 调用要有超时 |

### 工具调用的常见坑

| 坑 | 表现 | 解法 |
|------|------|------|
| 幻觉调用 | Agent 调了不存在的工具 | 严格定义工具列表 |
| 参数错误 | 传了错误的参数类型 | 参数校验+重试 |
| 循环调用 | Agent 反复调同一个工具 | 设置最大调用次数 |
| 过早调用 | 还没理解意图就调工具 | 先做意图识别 |
| 结果误用 | 工具返回了结果但 Agent 用错了 | 工具返回附带使用说明 |

| 框架 | 定位 | 适合谁 |
|------|------|--------|
| Coze（扣子） | 零代码 Agent 开发平台 | 产品经理、非技术人员 |

**产品判断**：框架选型是产品问题。问三个问题：谁在用？要上线还是做 Demo？需要多 Agent 吗？

## 2.9 Agent 安全与护栏

### 常见安全风险

| 风险 | 说明 | 防护 |
|------|------|------|
| Prompt 注入 | 用户输入覆盖了系统指令 | 输入过滤、角色分离、指令优先级 |
| 越狱 | 用户诱导 Agent 绕过限制 | 多层防护、输出审核 |
| 数据泄露 | Agent 输出中包含其他用户数据 | 记忆隔离、输出脱敏 |
| 幻觉 | Agent 编造不存在的事实 | RAG 提供事实锚点、输出验证 |
| 滥用 | 用户用 Agent 做违规操作 | 使用限制、内容审核 |

### 护栏设计层次

```
第一层：输入过滤 → 第二层：指令约束 → 第三层：工具限制 → 第四层：输出审核 → 第五层：人工兜底
```

**产品判断**：安全是产品设计的一部分。产品经理需要定义什么场景转人工、输出审核标准是什么、用户数据在链路中经过哪些节点。

# 第三篇：AI 产品经理角色与能力

## 3.1 AI PM vs 传统 PM

| | 传统 PM | AI PM |
|------|---------|-------|
| 核心产出 | 功能需求文档、原型、PRD | 上述 + 评测集、Prompt 策略、Badcase 分类 |
| 技术边界感 | 懂技术原理即可 | 必须懂 AI 能做什么、不能做什么、成本多少 |
| 不确定性 | 需求相对确定，效果可预期 | 模型输出不确定，效果需要持续迭代 |
| 迭代方式 | 上线→看数据→改功能 | 上线→看 Badcase→改 Prompt/链路→重新评测 |
| 成功标准 | 用户增长、留存、转化 | 上述 + 任务完成率、准确率、幻觉率 |
| 协作对象 | 工程师、设计师、运营 | 上述 + 算法工程师、标注团队 |

### AI PM 不需要 vs 需要

**不需要**：手写模型训练代码、自己训模型、会调 CUDA、看懂 Transformer 数学公式。

**需要**：知道这些概念是干什么的能和技术团队对话、能判断什么场景该用什么技术方案、能设计评测方案。

## 3.2 AI 产品经理的核心能力

### 技术边界感

知道当前 AI 技术能做什么、不能做什么。

**能做什么**：文本理解与生成、信息检索与总结、代码辅助、翻译、分类、情感分析、简单推理。

**不能做什么**：精确数学计算、100% 事实准确、真正的推理（不是统计续写）、理解物理世界、保证不产生幻觉。

**面试体现**：面试官问"你怎么用 AI 解决 XX"，你能说出"这个场景 LLM 搞不定，因为 XX"——这比"我可以用 Agent 做"更能证明边界感。

### 评测体系设计

不是"感觉好不好用"，而是能设计可量化、可复用的评测标准。

**核心能力**：定义 MECE 评测维度、设计评测集（样本来源、数量、标注方式）、选择评测执行方式、从 Badcase 反推产品改进。

**面试体现**：不要说"我们提高了准确率"，要说"用 30 道 Golden Set，五维评分，Spearman 校准后一致性从 0.70 到 0.85"。

### Prompt Engineering 产品化

不是"写一段 prompt"，而是管理 prompt 的完整生命周期。

**核心能力**：设计 Prompt 结构（分层解耦）、迭代 Prompt（基线→修改→评测→对比→决定）、A/B 测试 Prompt、管理 Prompt 版本。

**面试体现**：不要说"我优化了 prompt"，要说"我定位到第三层任务指令的评分规则缺少岗位锚定，增加了 5 条规则后，岗位匹配度从 2.1 到 4.3"。

### Badcase 驱动迭代

不是"上线了就算完"，而是建立发现→诊断→修复→验证的闭环。

**核心能力**：从用户反馈/评测数据中发现 Badcase、诊断根因（是 Prompt 问题、检索问题、还是链路问题）、给出修复方案、用同一套评测集验证。

**面试体现**：讲一个具体的 Badcase，说清楚现象、诊断过程、修复动作、验证结果。不要只说"改了一下 prompt 就好了"。

### 数据敏感度

不是"会写 SQL"，而是能定义数据口径、判断数据质量。

**核心能力**：每个指标都知道样本量、定义、口径、基线；能判断数据异常是真实问题还是统计偏差；能从数据中发现问题，而不是被数据带着走。

**面试体现**：提到的每个数字都能说出"样本量 30 题、人工标注为基准、Spearman 校准"——这就是数据敏感度。

## 3.3 AI 产品经理的常见岗位类型

| 岗位 | 核心工作 | 适合谁 |
|------|---------|--------|
| AI 应用 PM | 用现有 AI 能力做产品 | 大多数 AI PM |
| AI 平台 PM | 做 AI 开发平台/工具 | 有技术背景的 PM |
| AI 策略 PM | 定义 AI 产品方向和安全边界 | 有经验的高级 PM |
| 大模型 PM | 参与模型训练和优化 | 有算法背景的 PM |

**实习生/校招**：90% 是 AI 应用 PM。面试官不指望你训过模型，但希望你能说清楚你做的产品里 AI 怎么用的、怎么评测的、怎么迭代的。

## 3.4 AI 产品经理的成长路径

```
实习生：做一个完整的 AI 项目，建立评测意识
  ↓
初级 PM：独立负责一个 AI 功能模块，有真实用户数据
  ↓
中级 PM：独立负责一个 AI 产品，有完整的评测和迭代体系
  ↓
高级 PM：定义 AI 产品方向，管理多个 AI 产品，搭建团队
```

### 实习生阶段该做和不该做的

**该做的**：1-2 个完整的 AI 项目（可以是个人项目）、有一套评测体系（哪怕只是 30 题 Golden Set）、有几个能讲清楚的 Badcase 迭代案例、能说清楚每个技术决策的"为什么"。

**不该做的**：简历上写"负责从 0 到 1 搭建 AI 平台"（实习生做不到）、罗列技术名词但不解释为什么用、虚构没有的数据。

# 第四篇：AI 产品设计方法论

## 4.1 核心原则

### 不确定性是默认状态

传统产品输入输出确定——点按钮就跳转。AI 产品不同——同一个 prompt 两次结果可能不同。这是本质，不是 bug。

**产品设计含义**：不要设计"AI 必须 100% 准确"的产品（做不到）；设计"AI 可以犯错但用户能纠正"的产品；给 AI 输出留"退路"——用户可以编辑、重试、跳过。

### 用户预期管理比模型能力更重要

用户对 AI 的预期往往高于实际能力。产品经理的首要任务是管理预期，而非提升模型。

**产品设计含义**：明确告诉用户 AI 能做什么不能做什么；AI 输出不确定时标注"仅供参考"或给出置信度；高风险场景明确提醒"请咨询专业人士"。

### 渐进式信任

用户对 AI 的信任是逐步建立的。

**产品设计含义**：第一阶段 AI 做辅助（建议、参考），用户自己决策；第二阶段 AI 做半自动（AI 执行，用户确认）；第三阶段 AI 做全自动（AI 执行，用户事后检查）。不要一上来就做全自动。

## 4.2 交互模式

| 模式 | 形态 | 代表产品 | 适合场景 | 不适合场景 |
|------|------|---------|---------|----------|
| 对话式 | 自然语言对话 | ChatGPT、Claude | 开放式任务、探索性场景 | 高频重复操作、精确输入 |
| 辅助式（Copilot） | 嵌入工作流 | GitHub Copilot、Notion AI | 用户已有明确工作流 | AI 主导决策 |
| 自主式（Agent） | AI 自主完成 | AutoGPT、Devin | 任务明确、出错成本低 | 需要人类判断、出错成本高 |
| 嵌入式 | AI 在幕后 | 搜索 AI 摘要、电商推荐 | AI 是增强，用户无感 | AI 是核心卖点 |

### 对话式设计要点

首轮对话给用户"引导"——告诉用户能问什么；多轮对话要有"记忆"——用户说过的不要重复问；错误处理要友好——AI 没理解时给用户一个"换个问法"的提示。

### 辅助式设计要点

AI 不要打断用户；AI 提供"建议"而非"替代"——用户可以选择接受或忽略；出错成本要低——一键撤销 AI 的修改。

### 自主式设计要点

必须有"中途干预"——用户可以在任何时候暂停、修改、取消；必须让用户"看到过程"——AI 在做什么，做到哪一步了；必须有"结果确认"——用户确认再执行不可逆操作。

## 4.3 错误处理设计

### 错误类型与应对

| 错误类型 | 表现 | 应对 |
|---------|------|------|
| 幻觉 | AI 编造了不存在的信息 | 标注来源、提供"核实"按钮 |
| 理解偏差 | AI 误解了用户意图 | 提供"换个说法"的引导 |
| 能力边界 | 用户要求 AI 做不到的事 | 明确告知能力边界 |
| 格式错误 | AI 输出格式不符合预期 | 重试按钮、格式自动修复 |
| 超时 | AI 响应太慢或卡住 | 超时提示、降级方案 |
| 工具调用失败 | AI 调用的外部 API 挂了 | 降级处理、告知用户、允许重试 |

### 错误处理四原则

1. **永远不要让用户看到"系统错误"**——把技术错误翻译成用户能理解的语言
2. **给用户一个"下一步"**——不要只说"出错了"，要告诉用户能做什么
3. **错误是学习机会**——每次错误都应该"学到"这个 case，下次不再犯
4. **降低错误的代价**——AI 输出如果能一键撤销、一键修改，用户对错误的容忍度会大幅提升

## 4.4 数据飞轮

```
用户使用 → 产生数据 → 优化模型 → 体验变好 → 更多用户使用 → 更多数据
```

| 环节 | 产品设计要点 |
|------|------------|
| 收集数据 | 用户点赞/踩、修改 AI 输出、复制/不复制、继续追问/放弃 |
| 分析数据 | 哪些 case 用户踩了？哪些场景用户放弃了？ |
| 优化模型 | 用 Badcase 数据迭代 Prompt/RAG/微调 |
| 验证效果 | 用同一套评测集验证，确认体验变好了 |

**常见陷阱**：没有收集数据、收集了不分析、分析了不行动、行动了不验证。

## 4.5 产品形态决策树

### 选什么产品形态

```
用户有明确的任务吗？
  ├─ 没有 → 对话式（让用户探索）
  └─ 有 → 任务可以自动化吗？
           ├─ 可以，出错成本低 → 自主式（Agent）
           ├─ 可以，出错成本高 → 辅助式（Copilot，AI 建议人确认）
           └─ 不可以 → AI 是增强还是核心？
                       ├─ 增强 → 嵌入式（AI 在幕后）
                       └─ 核心 → 对话式 + 辅助式混合
```

### 选什么 AI 技术方案

```
任务需要外部知识吗？
  ├─ 需要 → 加 RAG
  └─ 不需要 → 纯 Prompt 够吗？
                ├─ 够 → 纯 Prompt
                └─ 不够 → 需要多步推理吗？
                          ├─ 需要 → Agent（规划 + 工具调用）
                          └─ 不需要 → 微调模型
```

## 4.6 AI 产品设计的常见错误

| 错误 | 表现 | 正确做法 |
|------|------|---------|
| 为 AI 而 AI | 不需要 AI 的地方硬塞 AI | 先问"不用 AI 能不能做" |
| AI 打头，体验垫底 | 能力强但交互烂 | 把 AI 输出包装好——格式清晰、延迟可控、错误友好 |
| 过度承诺 | 写"100% 准确" | 说"AI 辅助分析，建议人工复核" |
| 没有评测就上线 | 感觉"差不多"就上线 | 上线前至少跑一轮离线评测 |
| 上线后不管了 | 上线庆祝，然后没人管 | 建立 Badcase 收集→分析→修复→验证闭环 |

# 第五篇：面试高频考点

## 5.1 必须能讲清楚的概念

- **RAG 的原理和适用场景**：检索→增强→生成三步，什么时候用 RAG 什么时候微调
- **Prompt Engineering 的产品化**：prompt 的生命周期管理——怎么写、怎么测、怎么迭代、怎么 A/B
- **评测体系设计**：不是"准确率"，是 MECE 维度、样本来源、评分标准、自动化执行、badcase 闭环
- **Agent 架构选型**：Fixed Workflow vs Sub-agent vs Agent Team，说清楚什么时候用哪个
- **幻觉控制**：不是"优化 prompt"，是 RAG 锚定、输出验证、人工兜底

## 5.2 必问题与参考答案

**"你做的 AI 产品和普通产品有什么区别？"**

> 核心区别在于不确定性管理。普通产品的输入输出是确定的——点按钮就跳转。AI 产品的输出不确定——同一个 prompt 两次结果可能不同。所以我的工作重心不是"做功能"，而是"建评测体系、管 Badcase 闭环、控制输出质量"。

**"你在这个项目里具体做了什么？"**

> 不要说"我负责产品设计"。要说：我定义了用户场景，拆解了产品链路，设计了评测体系，定位了 Badcase 并修复。每个动作都要有具体的技术名词和量化结果。

**"你怎么衡量 AI 产品做得好不好？"**

> 三个层次：基础层——格式正确率、响应时间、幻觉率；任务层——意图理解准确率、检索命中率、任务完成率；用户层——满意度、留存率、复访率。我在第二个层次，用 Golden Set 做离线评测。

**"你做的 Agent 和普通 Chatbot 有什么区别？"**

> 普通 Chatbot 是一问一答，无状态。我的 Agent 有意图识别、多步推理、工具调用能力。比如用户说"帮我分析竞品"，Agent 会拆解成搜索、读取、对比、输出四步，而不是直接编一个回答。

**"你怎么保证 Agent 输出的质量？"**

> 三条线：事前——RAG 提供事实锚点，Prompt 约束输出格式和边界；事中——LLM-as-a-Judge 实时评分；事后——Badcase 归因到具体链路节点，迭代修复后用同一评测集验证。

**"RAG 和微调有什么区别？什么时候用哪个？"**

> RAG 适合知识频繁更新、需要可解释性、有明确知识库的场景。微调适合需要模型学会某种风格、推理模式、或特定领域语言习惯的场景。RAG 改的是"输入"，微调改的是"模型本身"。成本和维护方式完全不同。

**"你的 Agent 如果出错了，你怎么排查？"**

> 沿链路逐节点排查：意图理解对不对？检索结果对不对？拼接的上下文对不对？Prompt 逻辑对不对？模型输出对不对？每个节点有明确的输入输出，出问题能定位到具体节点，而不是笼统说"模型答错了"。

**"LLM 的幻觉问题你怎么解决？"**

> 三层防护：第一层，RAG 提供事实锚点，让模型基于检索到的真实信息回答；第二层，输出验证，用 LLM-as-a-Judge 检查输出中是否有与检索结果矛盾的内容；第三层，高风险场景转人工。幻觉不可能完全消除，产品经理的职责是把它控制在可接受范围内。

**"你怎么控制 API 成本？"**

> 四个手段：缓存高频问题的结果；意图识别前置，简单问题走正则不调 LLM；分层模型，简单任务用小模型，复杂任务用大模型；控制上下文长度，不把无关信息塞进 prompt。

## 5.3 进阶题与参考答案

**"你对 AI 产品经理这个岗位的理解是什么？"**

> 传统 PM 的核心能力是需求分析、产品设计、项目管理。AI PM 多了三个：技术边界感（知道 AI 能做什么不能做什么）、评测体系设计（能定义什么叫"好"）、Badcase 驱动的迭代能力（不是上线就完了，是持续发现和修复问题）。本质上，AI PM 是"不确定性管理者"。

**"你觉得 AI 产品最难的是什么？"**

> 不是技术，是预期管理。用户以为 AI 什么都能做，但实际上 AI 会犯错、会幻觉、会不稳定。产品经理要做的不是让 AI 不犯错（做不到），而是让用户在 AI 犯错时能理解、能纠正、能继续用。

**"你怎么看 AI 产品经理会不会被 AI 取代？"**

> 执行层会被取代——写 PRD、画原型这些 AI 已经能做。但判断层不会被取代——定义什么场景该用 AI、选择什么技术方案、设计评测标准、判断 Badcase 优先级、和 stakeholder 沟通取舍。这些需要的是判断力，不是执行力。

**"你设计产品时怎么考虑 AI 的不确定性？"**

> 三个手段：一是限制 AI 的输出范围——约束在结构化输出里；二是给用户"退路"——AI 输出错了用户可以一键修改、重试、跳过；三是分级使用——低风险场景 AI 自动处理，高风险场景转人工确认。

**"你怎么说服团队用 AI 而不是传统方案？"**

> 先算账。AI 方案的成本（API 费用、延迟、准确率）和传统方案的成本（开发时间、维护成本、规则复杂度）对比。如果 AI 方案在成本可接受范围内，能解决传统方案解决不了的问题（如语义理解、开放式问答），就值得用。

**"你觉得 AI 产品的护城河是什么？"**

> 不是模型本身——模型是商品，大家都能用。护城河是：数据飞轮（用户越多数据越多模型越好）、评测体系（你能精确衡量好坏，竞品靠感觉）、垂直场景的理解（通用 AI 不知道你的用户要什么）。技术可以复制，但你对用户场景的理解和你的数据飞轮复制不了。

## 5.4 面试防御话术

**当面试官质疑你的项目"太简单"**

> 这个项目看起来简单，但每个节点都有产品判断。比如意图识别我选了正则+LLM 兜底，没用 BERT 微调，因为现阶段意图类型只有 5 种，正则够用。架构上我选了 Fixed Workflow，没上 Agent Team，因为面试评分场景链路固定。每个技术选型我都能说出为什么。

**当面试官质疑你的数据"样本太小"**

> 30 题确实不算大，但这是我的 Golden Set 而不是上线数据。Golden Set 的价值不是样本量大，是每道题都有人工标注的黄金标准，能精确衡量每次迭代的效果。30 题的五维评分对标 30 道题的人工标注，这是一套完整的评测闭环。

**当面试官质疑"你只是调了 Prompt"**

> 调 Prompt 只是动作，重点是调之前的诊断和调之后的验证。我定位到问题是第三层任务指令缺少岗位锚定逻辑，不是笼统地"改了一下 prompt"。改完用同一套 Golden Set 验证，岗位匹配度从 2.1 到 4.3。这个过程和传统 PM 的"发现需求→设计方案→上线验证"没有本质区别，只是我的"代码"是 Prompt。
